## Слайд 3 · Архитектура

*[На экране: схема ill-arch, легенда внизу]*

Как устроено технически. Это **RAG-pipeline**, не «ChatGPT с файлами».

Слева на схеме — **Telegram и Web**. Дальше **Orchestrator** — наш FastAPI на сервере. Он не модель, а **дирижёр**: координирует цепочку агентов.

**Qdrant + e5** — семантический поиск по docx. **LLM** — только генерация ответа по **найденным** фрагментам. На demo — **GigaChat API**, в production — **Ollama** локально, закрытый контур.

**Evaluator** — confidence: если match слабый, не отвечаем. **Escalation** — оператор и тикет. **Analytics** — метрики и weak spots в **Redis**.

Документы — файлы на диске. Поиск — **Qdrant**. Сессии и analytics — **Redis**. Картинки из docx — отдельное хранилище.

→ *следующий слайд*

# Если спросят — по порядку ответа

**LLM на одном сервере или orchestrator?**  
Orchestrator — FastAPI **у нас**. LLM — шаг Generator: demo **GigaChat в облаке**, prod **Ollama локально**. Это разные роли.

**Почему не врёт?**  
Confidence — score похожести вопроса на фрагмент в Qdrant. Ниже **0.45** — оператор, LLM не выдумывает. Пример: «график офис-менеджера» — score 0 → эскалация.

**RAG без переобучения?**  
Docs в Qdrant, не в модели. Обновили docx → reindex — готово.

**Картинки в docx?**  
Извлекаем PNG → vision описывает → e5 → Qdrant → при visual-вопросе отдаём **оригинал** из docx.

**Где базы?**  
Qdrant — поиск. Redis — сессии, оператор, analytics. Docx — на диске.

**Одна фраза:** DocHelper — RAG по docx с источником, confidence против галлюцинаций, 94% на брифе, live @SUP_BARS_BOT.
