## 1. Соответствие решения поставленной задаче (0–5)

Задача кейса Барс Груп — **разгрузить первую линию поддержки**: сотрудники задают типовые вопросы по БАРС-Офису и корпоративным сервисам, хотя ответы уже есть в docx, но они не ищут их сами.

Мы сделали **DocHelper** — AI-агент первой линии, который:
- отвечает **только из корпоративной документации**, с указанием источника;
- работает **24/7** в Telegram и Web;
- при низкой уверенности или отсутствии ответа в docs — **эскалирует к оператору** с историей диалога;
- даёт **аналитику** для поддержки и техписов — где documentation не покрывает тему.

Метрика брифа: **≥ 40% автоматизации**. Факт: **94%** — 16 из 17 кейсов, воспроизводимый прогон `test_brief_cases.py`, отчёт TEST_REPORT.md.

Мы закрыли не «чат с GPT», а **именно задачу L1**: типовые вопросы — бот, сложные и непокрытые — человек, без галлюцинаций.

**Фраза для жюри:** «DocHelper напрямую решает задачу кейса — автоматизация первой линии по docs Барс с эскалацией и источником. 94% на брифе при цели 40%.»

## 2. Оригинальность — нестандартные подходы (0–5)

Стандартный подход — «загрузить PDF в ChatGPT». Мы пошли иначе:

**Orchestrated RAG pipeline** — шесть специализированных агентов под одним FastAPI, а не один chatbot.

**Распознавание знаний в картинках docx** — главная боль Барс: ценности компании, скриншоты интерфейса лежат в PNG внутри Word. Мы извлекаем `word/media/`, vision-модель описывает иллюстрации, OCR-текст для слайдов «Ценности», индексируем в Qdrant — RAG находит то, чего нет в «голом» тексте абзацев.

**Confidence gate** — бизнес-правило «авто или оператор», а не надежда на LLM. Кейс №6: «график офис-менеджера» — в docs нет → эскалация, не выдумка.

**Dual LLM** — GigaChat на demo, Ollama в prod; embeddings, Qdrant, Whisper — локально.

**Analytics как обратная связь** — weak spots показывают, куда дописывать Confluence, а не просто «бот ответил».

**Фраза для жюри:** «Мы не скопировали chatbot — мы решили проблему текста в картинках docx и построили pipeline с gate на уверенность и analytics для техписов.»

## 4. Аргументация выбранного решения (0–5)

**Почему RAG, а не fine-tune?** Документы Барс **обновляются** — reindex docx за минуты, без переобучения модели.

**Почему не «просто ChatGPT с файлами»?** Нет контроля источника, нет confidence gate, риск галлюцинаций. Нам нужен **audit trail** — откуда ответ, какой docx.

**Почему confidence + эскалация?** L1 не должна врать. Если в docs нет — оператор. Кейс №6 доказывает правильность подхода.

**Почему vision для картинок в docx?** Без этого не работают вопросы про ценности и интерфейс — 40% брифа провалились бы.

**Почему GigaChat + Ollama?** Demo — качество и скорость русского; prod — закрытый контур Барс, данные не уходят наружу.

**Почему Telegram + Web?** TG — с телефона за 5 секунд; Web — корпоративный чат + operator + analytics.

**Фраза для жюри:** «Каждый выбор — под ограничения enterprise: актуальные docs, источник, no hallucination, closed loop analytics.»

**Фраза для жюри:** «Три роли — три интерфейса: сотрудник в TG/Web, оператор в panel, жюри — presentation с live QR.»

## 7. Использование AI-агентов (0–5)

DocHelper — **orchestrated pipeline из 6 AI-агентов**, не один LLM:

1. **Speech** — Whisper: голос → текст  
2. **Retriever** — e5 + Qdrant: семантический поиск, rerank  
3. **Generator** — GigaChat/Ollama: ответ строго по контексту  
4. **Evaluator** — confidence: авто vs оператор  
5. **Escalation** — тикет, очередь, TG support chat  
6. **Analytics** — логирование, weak spots, тренды  

Плюс **Vision-agent** при индексации и на скриншотах пользователя.

Orchestrator в FastAPI координирует цепочку: rate limit → speech → retriever → generator → evaluator → escalation, параллельно analytics.

**Фраза для жюри:** «Шесть именованных агентов с разными задачами под orchestrator — это AI-agent architecture, не wrapper над ChatGPT.»
